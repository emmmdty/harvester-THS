# 抖音内容分级分类维护文档

Prompt 版本：`douyin-taxonomy-v2026-06-10`

本标准用于飞书“抖音渠道”表的 `一级类型`、`二级类型` 两列，以及后续 text-only 基线分类和 MiniMax sampled-media 多模态分类。分类标签采用短标签：输出字段不写括号、斜杠和“类”字，解释写在本文档中。

## 输出规则

- `platform` 固定为 `douyin`。
- `primaryType` 只能是：`股友说`、`财商动画`、`图文`、`社区话题`、`说唱`、`长视频`、`盘点`。
- `secondaryType` 只能从对应一级类型下选择；`说唱`、`长视频` 的 `secondaryType` 必须为空字符串。
- 模型不得创造新标签，不得把“资讯”“理财内容”“大佬采访”等旧内容类型写入 `primaryType` 或 `secondaryType`。
- `confidence` 为 0 到 1 的数字；低于 0.75、证据冲突、素材缺失或输出非法时，`reviewStatus` 写 `需人工复核`。
- `reason` 用一句中文解释判断依据，必须引用标题、tag 或画面中的真实线索。
- `text-only` 只作为低成本基线或 A/B 对照，使用标题、tag、账号、作品类型和旧内容类型。
- `sampled-media` 必须拿到视频抽帧、图文图片或页面截图后才调用 MiniMax；素材缺失时标记 `素材获取失败需复核`，不伪装成多模态，也不自动降级成文本分类。
- v1 不配置、不调用 ASR/OCR；后续只有确认可运行后端后再加入。

统一 JSON：

```json
{
  "platform": "douyin",
  "primaryType": "",
  "secondaryType": "",
  "confidence": 0,
  "reason": "",
  "evidence": [],
  "assetSignals": [],
  "reviewStatus": "通过"
}
```

## 易混边界

- `股友说` vs `社区话题`：核心是股民身份洞察、交易心态、悟道观点时归 `股友说`；核心是评论区互动、话题提问、段子调侃或产品种草时归 `社区话题`。
- `图文` vs `盘点`：形式是图文但核心为理财认知、人物故事或人生话题时归 `图文`；核心为资金、行情、榜单、行业、产业链、股票清单时归 `盘点`。
- `财商动画` vs `股友说`：画面是动画、故事化表达且主题是财商选择或成长磨难时归 `财商动画`；纯口播股民观点不要仅因出现“股民”归入动画。
- `说唱`、`长视频` 是形式优先标签。只要形式明确，应优先归入对应一级类型，二级留空。

## MiniMax 多模态提示词

适用场景：已拿到视频抽帧、图文图片、页面截图或完整素材时使用。素材无法获取或素材处理失败时，标记 `素材获取失败需复核`，不调用 MiniMax sampled-media。

```text

唯一分类依据是用户提供的本地分类标准，不允许引入外部抖音垂类口径。
请先判断 primaryType，再只在该 primaryType 下选择 secondaryType。
只能返回 JSON，不要返回 Markdown、解释或额外文本。

输出 JSON：
{
  "platform": "douyin",
  "primaryType": "",
  "secondaryType": "",
  "confidence": 0,
  "reason": "",
  "evidence": [],
  "assetSignals": [],
  "reviewStatus": "通过|需人工复核"
}

字段规则：
- primaryType 只能是：股友说、财商动画、图文、社区话题、说唱、长视频、盘点。
- secondaryType 只能从对应一级类型下选择；说唱、长视频必须为空字符串。
- evidence 引用标题、tag 或画面中的短片段。
- assetSignals 写画面/图文素材提供的判断线索；没有则返回空数组。
- confidence 低于 0.75、证据冲突、素材看不清或标签边界不确定时，reviewStatus 写 需人工复核。

输入字段：

使用本文档“分类表”和“易混边界”。
```

## Text-only 基线提示词

适用场景：低成本基线、A/B 对照，或明确决定不启用多模态时使用。不要把 text-only 结果当作 sampled-media 结果；标题和 tag 不足以支持分类时必须进入复核。

```text
你是抖音渠道内容分级分类助手。
只根据标题和 tag词 判断一级类型、二级类型；标题和 tag 证据不足时返回需人工复核，不要猜测素材形态或画面内容。
不得使用外部知识补全素材内容，不得创造新标签。
只能返回 JSON，不要返回 Markdown、解释或额外文本。

输出 JSON：
{
  "platform": "douyin",
  "primaryType": "",
  "secondaryType": "",
```

## 人工反馈列

人工纠正直接维护在飞书表，不先做独立后台。建议在 AI 输出列后追加以下列：

| 列名 | 可选值/填写方式 | 用途 |
| --- | --- | --- |
| 人工分类状态 | `正确`、`已纠正`、`边界待定`、`不纳入样本` | 标识这条 AI 分类能否进入反馈样本。 |
| 人工一级类型 | 抖音一级短标签 | `已纠正` 时必填；`正确` 时可留空。 |
| 人工二级类型 | 对应一级下的二级短标签，或空 | `已纠正` 时按最终判断填写。 |
| 人工备注 | 自由文本 | 记录错误原因、边界说明或不纳入原因。 |

处理规则：

- `正确`：AI 标签可信，可作为金标样本。
- `已纠正`：人工一级/二级覆盖 AI 标签，可作为错误样本和金标样本。
- `边界待定`：暂不进入 prompt 和评测集，等业务定口径。
- `不纳入样本`：素材缺失、标题污染、重复异常、非正常业务内容，不进入反馈样本。

## Prompt 反馈进化流程

1. 每周导出人工反馈行，保留标题、tag、素材证据、AI 输出、人工输出、人工备注、Prompt 版本。
2. 将 `正确` 和 `已纠正` 行组成回放集；`边界待定` 和 `不纳入样本` 不参与回放。
3. 同类错误累计 3 条以上，或影响关键业务分类时，才修改分类标准、边界规则或少样本示例。
4. 修改 Prompt 前后必须用同一批回放集对比：总准确率、一级准确率、二级准确率、低置信复核率、旧样本回归错误。
5. 版本号按日期更新，例如 `douyin-taxonomy-v2026-06-10`；变更记录必须写清：变更原因、采纳样本、修改内容、回放结果、是否影响旧分类。

## 后续新增分类流程

新增一级或二级标签时按以下顺序处理：

1. 收集至少 5 条代表样本，包含标题、tag、素材证据和期望分类。
2. 写清新增标签定义、判断线索、正例、反例和易混边界。
3. 检查是否能并入现有标签；只有现有标签无法稳定覆盖时才新增。
4. 更新本文档分类表和对应 Prompt 版本。
5. 回放旧样本，确认新增标签没有大量抢占旧标签。
6. 工程实施阶段再同步更新 taxonomy 代码、校验逻辑、下拉选项和测试。
